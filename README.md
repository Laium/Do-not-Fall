# Do Not Fall

> A 2D precision platformer built from scratch in Unity and C#, developed as a personal challenge to learn a new language and engine. The project demonstrates object-oriented programming principles through modular game systems, and reflects an iterative development process of problem-solving, debugging and refining mechanics to produce a fully playable experience.

---

## About This Project

Ascent is a personal project I built to push the boundaries of what I could teach myself. Coming in with no prior experience in Unity or C#, I wanted to challenge myself to take an idea from nothing — no templates, no tutorials followed step-by-step — and turn it into a complete, playable game.

The game is a 2D precision platformer inspired by Jump King, where the player must carefully navigate vertical levels, charging and releasing jumps with precision to climb higher. Fall too far, and you lose your progress. It is a simple concept, but one that demanded a deep understanding of physics, input handling and game state management to get right.

This project was never about building something polished for an audience. It was about proving to myself that I could independently design, build, debug and ship something from scratch — and learning everything I needed to along the way.

## Built With

- **Unity** — game engine
- **C#** — primary programming language

## Features

- Main menu with game launch and quit functionality
- Precision-based character movement with jump charge mechanics
- Collision detection and fall logic
- Multi-stage level design with progressively difficult platforming sequences
- In-game death counter UI to track player attempts

---

## Development Approach

One of the most important decisions I made early on was to treat this project the same way I would approach a software engineering task — with structure and intent, not just trial and error.

I applied **object-oriented programming principles** throughout, breaking the game into modular, reusable components: separate scripts for player input, movement physics, game state, and UI. This made the codebase easier to reason about and debug as complexity grew.

**Problem-solving and debugging** were central to the process. Getting the jump mechanic to feel right, for example, required multiple iterations — tweaking physics values, identifying edge cases in collision detection, and testing how small changes affected the overall feel of the game. Each bug was an opportunity to better understand how Unity's systems worked under the hood.

Building **from scratch** was a deliberate choice. It forced me to develop genuine understanding rather than relying on existing solutions, and gave me a much clearer picture of what it actually takes to take a project from an idea to a finished product.
